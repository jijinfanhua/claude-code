## 1. Agent 工作负载调研与分析

### 1.1 Agent 工作负载分类

通过对当前主流 AI Agent 平台（Claude Code、Cursor、Devin、SWE-agent、OpenHands、Codex、AutoGPT、BabyAGI、CrewAI 等）的深入调研，Agent 工作负载可分为以下六大类别：

#### 类别一：编码 Agent (Coding Agent)

| 属性 | 描述 |
|------|------|
| **代表产品** | Claude Code (Anthropic), Cursor Background Agent, Devin (Cognition), GitHub Copilot Workspace, Amazon CodeWhisperer Agent, Codex (OpenAI) |
| **工作模式** | 远程调用 LLM API → 本地执行工具调用（编译、测试、lint、git 操作） |
| **典型 CPU 需求** | 1-8 vCPU / 实例（Claude Agent SDK 推荐最小 1 vCPU） |
| **典型内存需求** | 1-16 GB / 实例（SDK 推荐最小 1 GiB） |
| **典型存储需求** | 5-30 GB / 实例（代码仓库 + 依赖） |
| **CPU 利用模式** | **高度突发性 (Bursty)**：LLM 思考时几乎空闲 (< 5% CPU)，工具调用时短暂飙升 (> 80% CPU) |
| **活跃/空闲比** | 约 20-40% 时间活跃（工具执行），60-80% 时间空闲（等待 LLM 响应或用户输入） |
| **网络需求** | 中等（LLM API 调用 + 包管理器下载） |
| **隔离要求** | 高（执行用户任意代码，需强隔离） |

#### 类别二：浏览器/Web Agent (Browser Agent)

| 属性 | 描述 |
|------|------|
| **代表产品** | Claude Computer Use, Anthropic Agent SDK (browser tool), Playwright-based agents, WebVoyager |
| **工作模式** | 启动无头浏览器 → 截图/DOM 分析 → LLM 决策 → 执行浏览器动作 |
| **典型 CPU 需求** | 2-4 vCPU / 实例（浏览器渲染引擎是 CPU 密集型） |
| **典型内存需求** | 4-8 GB / 实例（Chromium 内存占用大） |
| **CPU 利用模式** | **中等突发**：页面加载/渲染时 CPU 较高，等待 LLM 时较低 |
| **特殊要求** | 需要图形渲染能力（Xvfb 或类似）、较大共享内存 |

#### 类别三：数据分析 Agent (Data Analysis Agent)

| 属性 | 描述 |
|------|------|
| **代表产品** | Code Interpreter (OpenAI), Claude Analysis Tool, Julius AI, Jupyter-based agents |
| **工作模式** | 运行 Python/R 脚本 → 处理数据集 → 生成可视化 |
| **典型 CPU 需求** | 2-16 vCPU / 实例（取决于数据规模） |
| **典型内存需求** | 4-32 GB / 实例（大数据集在内存中处理） |
| **CPU 利用模式** | **持续高负载阶段性**：数据处理期间 CPU 持续 70-100%，空闲期间 < 5% |
| **特殊要求** | 可能需要较长运行时间，内存是主要瓶颈 |

#### 类别四：多 Agent 编排 (Multi-Agent Orchestration)

| 属性 | 描述 |
|------|------|
| **代表产品** | CrewAI, AutoGen, LangGraph, MetaGPT, Claude Swarm |
| **工作模式** | 多个 Agent 协作：规划 Agent → 编码 Agent → 审查 Agent → 测试 Agent |
| **典型 CPU 需求** | 16-32 vCPU / 管道（3-6 并发 Agent） |
| **典型内存需求** | 32-64 GB / 管道 |
| **CPU 利用模式** | **波浪式**：不同 Agent 交替活跃，整体利用率相对平稳 |
| **特殊要求** | Agent 间通信延迟敏感 |

#### 类别五：自主 Agent / 后台 Agent (Autonomous Agent)

| 属性 | 描述 |
|------|------|
| **代表产品** | Devin (长时间运行), Claude Code headless mode, SWE-agent, OpenHands |
| **工作模式** | 无人值守执行复杂任务，可能运行数小时 |
| **典型 CPU 需求** | 4-8 vCPU / 实例 |
| **典型内存需求** | 8-16 GB / 实例 |
| **CPU 利用模式** | 与编码 Agent 类似但持续时间更长，突发更频繁 |
| **特殊要求** | 需要状态持久化、长连接支持 |

#### 类别六：轻量级工具调用 Agent (Lightweight Tool-Calling Agent)

| 属性 | 描述 |
|------|------|
| **代表产品** | 各类 MCP Server, Function Calling agents, API orchestration agents |
| **工作模式** | 执行 HTTP 调用、数据库查询、文件操作等轻量工具 |
| **典型 CPU 需求** | 0.25-1 vCPU / 实例 |
| **典型内存需求** | 256 MB - 1 GB / 实例 |
| **CPU 利用模式** | **极度突发**：99% 时间空闲，1% 时间短暂活跃 |
| **特殊要求** | 最需要快速启动、最适合超卖 |

## 2. 工作负载资源画像总结

### 2.1 核心发现

通过上述调研，Agent 工作负载具有以下显著特征：

**特征一：极端突发性 (Extreme Burstiness)**
- Agent 工作负载的 CPU 利用呈现典型的 "脉冲" 模式
- 等待 LLM API 响应期间 CPU 接近零利用（占 60-80% 时间）
- 工具执行期间 CPU 短暂飙升至 80-100%（占 20-40% 时间）
- 峰均比高达 15-20x，这是超卖的最大机会

**特征二：高度异构性 (High Heterogeneity)**
- 不同类型 Agent 资源需求差异达 20x 以上
- 从轻量级工具调用 (0.25 vCPU) 到数据分析 (16 vCPU)
- 同一类型 Agent 不同任务间资源差异也达 1.8x

**特征三：内存是并发瓶颈**
- 内存不可压缩，超卖风险高于 CPU
- 内存消耗不可预测（15.4x 峰均比）
- CPU 是可压缩资源，更适合超卖

**特征四：快速启停是关键**
- Agent 实例生命周期通常为分钟到小时级
- 高并发场景要求秒级甚至毫秒级启停
- 池化预热是达到 50ms 目标的必要条件

### 2.2 资源需求矩阵（归一化）

| 工作负载类型 | CPU 需求 | 内存需求 | 启动频率 | 生命周期 | 超卖适合度 | 占比预估 |
|-------------|---------|---------|---------|---------|-----------|---------|
| 编码 Agent | 中 (1-8) | 中 (1-16G) | 高 | 中 | **极高** | 45% |
| 浏览器 Agent | 中 (2-4) | 高 (4-8G) | 中 | 短 | 高 | 10% |
| 数据分析 Agent | 高 (2-16) | 高 (4-32G) | 低 | 长 | 中 | 10% |
| 多 Agent 编排 | 高 (16-32) | 高 (32-64G) | 低 | 长 | 中 | 5% |
| 自主 Agent | 中 (4-8) | 中 (8-16G) | 中 | 长 | 高 | 15% |
| 轻量工具调用 | 低 (0.25-1) | 低 (256M-1G) | 极高 | 极短 | **极高** | 15% |

### 2.3 设计目标约束

| 目标 | 指标 |
|------|------|
| 沙箱启动耗时 | < 150ms（冷启动），池化后 < 50ms |
| 并发拉起能力 | 每分钟万级实例 |
| 单服务实例规模 | 10 万实例 |
| 调度吞吐 | 单集群 200 QPS |
| 资源池规模 | 百万核级别 |
| 单实例创建开销 | < 200ms，池化后 < 50ms |
| 性能目标 | 最高性能 + 最高性价比 + 低能耗 + 可超卖 |

## 3. 单 CPU 选型 Technical Proposal

### 3.1 选型方法论

Agent 沙箱工作负载对 CPU 的核心需求可概括为：

1. **核密度优先**：单 socket 核数越多 → 每台物理机承载实例越多 → TCO 越低
2. **能效比优先**：Performance/Watt 而非 Peak Performance
3. **内存带宽充足**：高并发实例共享内存带宽不成为瓶颈
4. **虚拟化性能**：硬件辅助虚拟化（VT-x/AMD-V/SVM）性能直接影响启动延迟
5. **缓存结构优化**：大量小实例对 L3 缓存争用敏感

### 3.2 候选 CPU 全面对比

#### 3.2.1 AMD EPYC 9005 系列 (Turin, Zen 5/5c)

| SKU | 核/线程 | 基频/睿频 | TDP | L3 缓存 | 内存通道 | 参考价格 | 核/瓦 |
|-----|--------|----------|-----|--------|---------|---------|-------|
| EPYC 9965 | 192/384 | 2.25/3.7 GHz | 500W | 384 MB | 12x DDR5 | $14,813 | 0.384 |
| EPYC 9755 | 128/256 | 2.7/4.1 GHz | 500W | 512 MB | 12x DDR5 | $12,984 | 0.256 |
| EPYC 9575F | 64/128 | 3.3/5.0 GHz | 400W | 256 MB | 12x DDR5 | - | 0.160 |
| EPYC 9135 | 32/64 | 3.2/4.3 GHz | 200W | 128 MB | 12x DDR5 | - | 0.160 |

**优势**：
- Zen 5c 紧凑核心（3nm）提供业界最高核密度 (192C/socket)
- 单 socket 即可承载 192 个 1vCPU 实例（未超卖），超卖后可达 600-1000+
- 12 通道 DDR5 提供超大内存带宽
- AMD-V + SEV-SNP 提供硬件级别安全虚拟化
- L3 缓存容量大（384 MB），每核 2 MB，适合多实例场景

**劣势**：
- 500W TDP 较高，需要高效散热
- Zen 5c 单核性能低于 Zen 5 标准核心
- 价格较高（~$15K/socket）

#### 3.2.2 Intel Xeon 6 Sierra Forest (E-core)

| SKU | 核/线程 | 基频/睿频 | TDP | L3 缓存 | 内存通道 | 核/瓦 |
|-----|--------|----------|-----|--------|---------|-------|
| Xeon 6980E | 288/288 | ~2.0/3.2 GHz | 350W | 108 MB | 8x DDR5 | 0.823 |
| Xeon 6780E | 144/144 | ~2.0/3.2 GHz | 250W | 108 MB | 8x DDR5 | 0.576 |
| Xeon 6710E | 64/64 | ~2.0/3.2 GHz | 185W | 96 MB | 8x DDR5 | 0.346 |

**优势**：
- **业界最高核/瓦比**：288 核仅 350W TDP，核/瓦达 0.823
- E-core (Crestmont) 专为吞吐型工作负载设计
- 2.5x 优于上代的机架密度
- 2.4x 优于上代的性能/瓦特
- 一项 VM 密度测试中，6930E(288E)稳定运行 900 个 2-vCPU KVM 虚拟机
- 功耗最低，3 年仅 16,300 kWh

**劣势**：
- 无超线程（SMT），288 核 = 288 线程（AMD 192 核 = 384 线程）
- L3 缓存较小（108 MB vs AMD 384 MB），每核仅 0.375 MB
- 8 通道 DDR5（vs AMD 12 通道），内存带宽受限
- 单核性能明显低于 P-core
- 生态系统相对较新

#### 3.2.3 Intel Xeon 6 Granite Rapids (P-core)

| SKU | 核/线程 | 基频/睿频 | TDP | L3 缓存 | 内存通道 | 核/瓦 |
|-----|--------|----------|-----|--------|---------|-------|
| Xeon 6980P | 128/256 | 2.0/3.8 GHz | 500W | 504 MB | 8x DDR5/MRDIMMs | 0.256 |

**优势**：
- 单核性能最强，适合延迟敏感场景
- 504 MB 超大 L3 缓存
- 支持 MRDIMMs，单 socket 最大 8TB 内存

**劣势**：
- 核密度不如 E-core 方案
- 500W TDP + 较少核心 = 核/瓦比最差
- 不适合 "大量小实例" 的 Agent 场景

#### 3.2.4 Ampere Altra / AmpereOne (ARM)

| SKU | 核/线程 | 基频 | TDP | 架构 | 核/瓦 |
|-----|--------|------|-----|------|-------|
| AmpereOne | 192/192 | 3.0 GHz | 250W | ARMv9 Neoverse | 0.768 |
| Altra Max | 128/128 | 3.0 GHz | 250W | ARMv8.2 Neoverse-N1 | 0.512 |

**优势**：
- **极致能效**：比 x86 能耗低 16-67%
- 每核独占流水线，无 SMT 争用，性能可预测
- NGINX 上 4x 计算效率优势，Memcached 2.66x 能效优势
- 适合 ARM 原生云工作负载
- 250W TDP 下 192 核，核/瓦 0.768

**劣势**：
- ARM ISA 兼容性问题：Agent 可能执行 x86 原生代码
- 生态系统不够成熟（部分开发工具/依赖库不支持 ARM）
- 缓存和内存带宽不如 AMD EPYC
- 对编码 Agent 来说，需要考虑编译工具链兼容性

### 3.3 综合评分与推荐

基于 Agent 工作负载特征，设计加权评分模型：

| 评估维度 | 权重 | AMD EPYC 9965 | Intel Sierra Forest 6980E | Intel Granite Rapids 6980P | Ampere AmpereOne |
|----------|------|---------------|---------------------------|---------------------------|-----------------|
| 核密度 (核/socket) | 25% | 9 (192C/384T) | **10** (288C/288T) | 6 (128C/256T) | 9 (192C/192T) |
| 能效比 (核/瓦) | 20% | 6 (0.384) | **10** (0.823) | 4 (0.256) | 9 (0.768) |
| 虚拟化性能 | 15% | **9** (AMD-V/SEV) | 8 (VT-x/TDX) | 8 (VT-x/TDX) | 7 (基础) |
| 内存带宽/容量 | 15% | **10** (12ch DDR5) | 7 (8ch DDR5) | 8 (8ch+MRDIMM) | 6 (8ch DDR5) |
| 缓存有效性 | 10% | **9** (384MB/2MB per core) | 5 (108MB/0.375MB) | **9** (504MB/3.9MB) | 5 |
| 生态兼容性 | 10% | **9** (x86 全兼容) | **9** (x86 全兼容) | **9** (x86 全兼容) | 5 (ARM) |
| 性价比 ($/核) | 5% | 7 (~$77/核) | 8 | 5 | **9** |
| **加权总分** | 100% | **8.35** | **8.65** | 6.65 | 7.35 |

### 3.4 推荐策略：混合选型 (Tiered CPU Strategy)

不建议单一 CPU 选型，而是根据工作负载类型分层部署：

#### Tier 1 — 高密度轻量层 (60% 机器)：Intel Xeon 6 Sierra Forest 6980E

- **适用负载**：轻量工具调用 Agent、编码 Agent（标准任务）、短生命周期沙箱
- **理由**：
  - 288 核提供最大实例密度
  - 0.823 核/瓦极致能效
  - 900 VM/node 的实测密度适合 Agent 场景
  - 用于池化预热沙箱（快速创建-执行-销毁循环）
- **单机承载**：超卖 5x 后可承载 ~1,400 个 1-vCPU Agent 实例

#### Tier 2 — 通用均衡层 (30% 机器)：AMD EPYC 9965 (192C)

- **适用负载**：编码 Agent（复杂任务）、浏览器 Agent、自主 Agent
- **理由**：
  - 384T 超线程在混合负载下利用率更高
  - 12 通道 DDR5 提供充足内存带宽
  - 384 MB L3 缓存减少多实例缓存争用
  - AMD SEV-SNP 提供硬件级租户隔离
- **单机承载**：超卖 5x 后可承载 ~960 个 1-vCPU Agent 实例

#### Tier 3 — 高性能计算层 (10% 机器)：AMD EPYC 9755 (128C) 或 EPYC 9575F (64C)

- **适用负载**：数据分析 Agent、多 Agent 编排、需要高单核性能的任务
- **理由**：
  - 更高单核频率（4.1GHz / 5.0GHz）适合 CPU 密集工具执行
  - 512 MB / 256 MB L3 缓存提供优秀的缓存命中率
  - 编译、数据处理等工具调用需要更强单核能力
- **单机承载**：超卖 3x 后可承载 ~384 / ~192 个 1-vCPU Agent 实例

### 3.5 CPU 特性需求 Checklist

无论选择哪款 CPU，以下特性必须支持：

| 特性 | 必要性 | 用途 |
|------|--------|------|
| 硬件虚拟化 (VT-x/AMD-V) | **必须** | microVM/KVM 加速 |
| 二级地址转换 (EPT/NPT) | **必须** | 虚拟内存性能关键 |
| 硬件安全虚拟化 (SEV/TDX) | 推荐 | 多租户安全隔离 |
| PCID/INVPCID | **必须** | 减少 TLB flush 开销（关键：影响 VM 切换性能） |
| IBRS/STIBP/SSBD | 推荐 | Spectre/Meltdown 缓解（安全但有性能开销） |
| AVX-512 / AVX2 | 推荐 | 加速编译、数据处理工具调用 |
| PMU (性能计数器) | 推荐 | 精准资源计量与调优 |

## 4. CPU 机架 Technical Proposal

### 4.1 机架设计目标

| 目标 | 指标 |
|------|------|
| 单机架承载实例数 | > 10,000 Agent 实例 |
| 单机架总核数 | > 2,000 物理核 |
| 单机架功耗预算 | 20-30 kW |
| PUE 目标 | < 1.15 |
| 机架可用性 | 99.95% |

### 4.2 推荐机架配置

#### 方案 A：高密度 Sierra Forest 机架（推荐作为主力）

```
┌─────────────────────────────────────────────────────┐
│              42U 标准机架 - Tier 1 高密度              │
├─────────────────────────────────────────────────────┤
│ U1-U2:   2x Top-of-Rack 交换机 (25/100GbE)         │
│ U3:      1x 机架管理/BMC 聚合单元                      │
│ U4-U39:  18x 2U 双路服务器                            │
│          每台: 2x Xeon 6980E (576核/node)            │
│          内存: 1 TB DDR5 (16x 64GB RDIMM)           │
│          存储: 2x 3.84TB NVMe SSD                   │
│          网络: 2x 25GbE + 1x 管理口                   │
│ U40-U41: 2x 液冷分配单元 (CDU)                       │
│ U42:     1x PDU/电力分配                              │
├─────────────────────────────────────────────────────┤
│ 机架总物理核数: 18 nodes × 576 cores = 10,368 物理核   │
│ 机架总内存: 18 nodes × 1 TB = 18 TB                  │
│ 机架总功耗: 18 × ~1,000W (双路) + 辅助 ≈ 22 kW       │
│ 超卖5x后承载实例: 10,368 × 5 = ~51,840 实例           │
│ PUE (直接液冷): ~1.08                                 │
```

**关键设计决策**：

1. **选择 2U 双路而非 1U 单路**：
   - 双路提供 576 核/node，减少节点数量从而降低管理开销
   - 2U 提供更好的散热空间和扩展性
   - 每节点内存总线独立，减少 NUMA 跨节点访问

2. **内存配置 1 TB/node**：
   - 基于 Agent 工作负载内存密集特征
   - 576 核 / 1TB = ~1.78 GB/核，匹配轻量 Agent 需求
   - 支持更高规格 Agent（编码/浏览器）同时运行

3. **NVMe SSD 配置**：
   - 2x 3.84TB 提供充足的本地存储用于 Agent 沙箱文件系统
   - 高 IOPS (>100K) 支持并发实例的 I/O 需求
   - RAID-1 保证数据可靠性

#### 5.3.2 调度器核心设计

**Cell Scheduler 设计（达成 200+ QPS 目标）**：

```
                    请求入队
                       │
                       ▼
            ┌─────────────────────┐
            │    Priority Queue   │  按优先级/SLA 排序
            │  (Lock-free MPMC)   │
            └──────────┬──────────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
    ┌─────┴─────┐ ┌───┴─────┐ ┌───┴─────┐
    │ Shard #1  │ │ Shard #2│ │ Shard #3│  并发调度分片
    │ Rack 1-17 │ │ Rack 18 │ │ Rack 35 │  (参考 Gödel/Omega)
    │           │ │   -34   │ │   -50   │
    └─────┬─────┘ └───┬─────┘ └───┬─────┘
          │            │            │
          └────────────┼────────────┘
                       │
                       ▼
            ┌─────────────────────┐
            │  Conflict Resolver  │  乐观并发冲突解决
            │  (CAS on Resource   │  (失败重试，通常 < 1%)
            │   State Store)      │
            └──────────┬──────────┘
                       │
                       ▼
                  Commit & Dispatch
```

**关键设计决策**：

1. **乐观并发调度（Omega 模式）**
   - 每个 Shard 维护节点资源的本地副本
   - 并发执行调度决策，最后通过 CAS 提交
   - 冲突率通常 < 1%（Agent 工作负载分布均匀）
   - 相比串行调度，吞吐提升 3-5x

2. **percentageOfNodesToScore 优化**
   - 900 节点的 Cell，设置评估比例为 10-15%
   - 仅评估 90-135 个候选节点即可找到足够好的放置点
   - 减少 85-90% 的评估开销

3. **调度吞吐计算**
   ```
   单 Shard 吞吐 = 1 / (评估延迟 + 提交延迟)
                  = 1 / (3ms + 1ms) = 250 QPS
   3 Shard 并发   = 250 × 3 × 0.85 (冲突损耗) = ~637 QPS
   远超 200 QPS 目标
   ```

#### 5.3.3 调度策略

针对 Agent 工作负载的差异性，设计多维调度策略：

| 策略 | 适用场景 | 实现方式 |
|------|---------|---------|
| **Tier 亲和调度** | 按 Agent 类型路由到对应 Tier 机架 | Label-based affinity |
| **Bin Packing (装箱)** | 提高利用率，释放空节点 | 优先调度到已有实例的节点 |
| **NUMA 感知** | 减少跨 NUMA 内存访问 | Topology Manager + CPU Pinning |
| **池预热均匀** | 预热池实例均匀分布 | Spread scheduling |
| **热点避让** | CPU 温度过高时迁移 | 热力学感知调度 |
| **突发预留** | 高优先级 Agent 保留资源 | Priority preemption |

**三级池化设计**：

| 池级别 | 技术实现 | 获取延迟 | 内存开销/实例 | 每节点数量 | 用途 |
|--------|---------|---------|-------------|-----------|------|
| Hot Pool | 完全启动的 microVM/容器，OS + 运行时就绪 | < 10ms | 128-256 MB | 20-50 | 高优先级/延迟敏感请求 |
| Warm Pool | Firecracker 快照 + CoW 内存映射 | < 50ms | 265 KB - 5 MB | 100-200 | 主要服务池（默认） |
| Cold Pool | 就绪的 rootfs/镜像，需 boot 或快照恢复 | < 150ms | ~0 (仅磁盘) | 不限 | 突发扩容 |

**关键技术：Firecracker 快照 + CoW Fork**

```
1. 模板准备 (一次性, ~15s):
   Boot Firecracker VM → 安装运行时 (Python/Node/Go) 
   → 预热常用库 → 创建快照 (内存 + CPU 状态)

2. 实例创建 (每次, <50ms):
   MAP_PRIVATE 映射快照内存 → Fork KVM VM 
   → 恢复 CPU 寄存器 → 应用增量配置 (网络/存储)
   → 实例就绪

3. 实例回收 (每次, <5ms):
   终止 VM 进程 → 释放 CoW 页面 → 回收网络/存储资源
```

**与 ZeroBoot 方案的对比**：

| 方案 | Spawn 延迟 | 内存/实例 | 成熟度 | 限制 |
|------|-----------|----------|--------|------|
| ZeroBoot CoW Fork | **0.79ms** p50 | ~265 KB | 原型阶段 | 单 vCPU、无网络 |
| Firecracker 快照恢复 | ~50ms | ~5 MB | **生产就绪** | 恢复时间略长 |
| 自研 CoW Fork + Firecracker | <5ms | ~1 MB | 需开发 | 需解决多 vCPU + 网络 |

**推荐方案**：短期使用 Firecracker 快照恢复（成熟可靠），中期借鉴 ZeroBoot 思路自研 CoW Fork 方案。

**关键设计**：

2. **状态存储选型**：

3. **心跳聚合**：每 10s 心跳，10 万实例 = 10,000 QPS 心跳负载，需要批量聚合

## 6. 超卖与弹性策略

### 6.1 CPU 超卖模型

Agent 工作负载天然适合 CPU 超卖，原因：

1. **CPU 是可压缩资源**：超卖只会导致性能降级，不会 OOM kill
2. **Agent 60-80% 时间空闲**：等待 LLM API 响应时 CPU 几乎零利用
3. **突发窗口短**：工具执行通常 < 10s，之后回归空闲

#### 6.1.1 超卖比设计

```
超卖比 = 虚拟核总分配 / 物理核总数

目标: 实现高效超卖而不显著影响用户体验
```

| 工作负载类型 | 推荐超卖比 | 理由 | 等效密度 |
|-------------|-----------|------|---------|
| 轻量工具调用 Agent | 8-10x | 99% 时间空闲 | 288 核 → 2,304-2,880 vCPU |
| 编码 Agent (标准) | 5-6x | 70% 时间空闲 | 288 核 → 1,440-1,728 vCPU |
| 编码 Agent (复杂) | 3-4x | 50% 时间空闲 | 192 核 → 576-768 vCPU |
| 浏览器 Agent | 3-4x | 渲染时 CPU 较高 | 192 核 → 576-768 vCPU |
| 数据分析 Agent | 2-3x | 计算密集期长 | 128 核 → 256-384 vCPU |
| 多 Agent 编排 | 2-3x | 多 Agent 交替活跃 | 128 核 → 256-384 vCPU |

**加权平均超卖比**（按工作负载占比）：

```
加权超卖比 = 0.15 × 9 + 0.45 × 5.5 + 0.10 × 3.5 + 0.10 × 3.5 + 0.10 × 2.5 + 0.10 × 2.5
           = 1.35 + 2.475 + 0.35 + 0.35 + 0.25 + 0.25
           = 5.025x

取整推荐: 全局平均超卖比 ≈ 5x
```

#### 6.1.2 超卖实现机制

**层级 1：cgroup v2 CPU Bandwidth + Weight**

```yaml
# 轻量工具调用 Agent (0.25 vCPU request, 2 vCPU limit)
cpu.weight: 25          # 相对权重 (默认 100 = 1 vCPU share)
cpu.max: "50000 100000" # 50ms/100ms = 0.5 CPU hard limit (burst 允许到 2x)

# 编码 Agent (1 vCPU request, 4 vCPU limit)  
cpu.weight: 100
cpu.max: "400000 100000" # 400ms/100ms = 4 CPU hard limit

# 数据分析 Agent (4 vCPU request, 8 vCPU limit)
cpu.weight: 400
cpu.max: "800000 100000" # 800ms/100ms = 8 CPU hard limit
```

**层级 2：CPU Burst (Linux 5.14+)**

```
启用 cpu.max.burst 允许实例在空闲期积累 CPU 配额
在工具调用突发时使用积累的配额，避免被限流

示例: 编码 Agent
- 正常 quota: 100ms/100ms (1 vCPU)
- burst 积累: 空闲 5s → 积累 500ms quota
- 工具执行时: 可突发使用 6 vCPU 持续 ~83ms，然后回归 1 vCPU
- 效果: 工具调用延迟显著降低，同时保持低平均资源消耗
```

**层级 3：动态超卖比调整**

```python
# 伪代码: 动态超卖比控制器
class DynamicOvercommitController:
    def adjust_overcommit_ratio(self, node_metrics):
        cpu_p95 = node_metrics.cpu_utilization_p95
        cpu_avg = node_metrics.cpu_utilization_avg
        throttle_rate = node_metrics.cfs_throttle_rate
        
        if cpu_p95 > 85 or throttle_rate > 5:
            # CPU 争用严重，降低超卖比
            self.reduce_overcommit(factor=0.8)
            self.trigger_evacuation(low_priority_instances)
        elif cpu_avg < 30 and throttle_rate < 0.1:
            # 利用率低，提高超卖比
            self.increase_overcommit(factor=1.2)
        
        # 时间维度动态: 工作时间超卖比低，非工作时间超卖比高
        if is_off_peak_hours():
            self.overcommit_ratio *= 1.5
```

### 6.2 内存管理策略

内存不可压缩，需要更保守的策略：

| 策略 | 实现 |
|------|------|
| 内存超卖比 | 1.5-2x（远低于 CPU） |
| KSM (Kernel Samepage Merging) | 合并相同内存页，Agent 基础镜像高度相似 |
| Balloon Driver | VM 内动态回收空闲内存 |
| 内存分级 | Hot: DRAM / Cold: PMEM 或 CXL 内存 |
| OOM 优先级 | 按 Agent 优先级设定 oom_score_adj |
| Eviction 阈值 | 节点内存 > 90% 时开始驱逐低优实例 |

### 6.3 性能保障机制

超卖环境下保障用户体验的关键机制：

#### 6.3.1 QoS 分级

| Burstable | 100-400 | Request 保证 | 中等 | 标准 Agent |

### 6.4 能效优化

| 策略 | 节能效果 | 实现方式 |
|------|---------|---------|
| 动态频率调节 (DVFS) | 15-25% | 低负载时降频至基频以下 |
| C-State 深度休眠 | 20-30% | 空闲核心进入 C6/C6N 状态 |
| Package C-State | 10-15% | 整个 socket 空闲时进入深度节能 |
| 节点级休眠 | 30-40% | 非高峰时段关闭整个节点 |
| 液冷自然冷却 | 70-95% 时间 | 无需冷机运行，大幅降低冷却能耗 |

### 7.4 风险与缓解

| 风险 | 影响 | 缓解方案 |
|------|------|---------|
| Sierra Forest 生态不成熟 | 兼容性问题 | 与 Intel 建立早期合作，提前验证 |
| 超卖导致性能波动 | 用户体验下降 | QoS 分级 + 动态超卖比 + SLA 监控 |
| 液冷泄漏 | 硬件损坏 | 泄漏检测传感器 + 自动关断 + 保险 |
